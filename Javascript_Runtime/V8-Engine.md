# JavaScript to Machine Code - V8 Engine Deep Dive

## Overview

This document provides a high-level walkthrough of everything that happens going from our human-friendly JavaScript file all the way down to something that computers can understand.

There are many parts to this process, but we'll focus on two things:
1. **Browser side of things**
2. **V8 side of things**

V8 is the JavaScript engine used in:
- Chromium-based browsers
- Node.js

## The Parser

### Loading the Script

Let's start from the beginning. We're trying to load a website that uses a small `calc.js` script.

As we're trying to load the website, the HTML parser encounters a `<script>` tag and tries to fetch the `calc.js` file from either:
- The network
- Cache
- A service worker that prefetched the file

### Byte Stream Decoder

Either way, a **stream of bytes** gets returned that gets sent to the **byte stream decoder**.

The decoder is part of the parser that takes care of:
- Decoding the stream of bytes
- Generating tokens based on the data received

**Example**: 
- It sees that the bytes decode to `f-u-n-c-t-i-o-n`
- It generates a token saying: "Hey, I know `function` is a keyword in JavaScript"
- Creates a token based on that
- Continues to do so for the rest of the stream

### Token Generation

As it's generating these tokens, it's actually sending them all down to the **parser**.

## Syntax Tree Generation

The parser goes ahead and creates **nodes** based on the tokens that match a certain syntax rule in JavaScript, for example:
- A variable declaration
- A function statement

Based on these nodes, the parser generates an **Abstract Syntax Tree (AST)** that represents our program.

**Note**: The real AST also contains extra information about a program, but this is a simplified view.

### Syntax Error Checking

While doing that, it's checking for **syntax errors** because:
- The tokens themselves may be valid
- But they may not actually match a certain syntax rule

## JavaScript Engine: V8 Ignition Interpreter

Finally, it's time for the JavaScript engine to do its work.

The AST is sent down to **V8's Ignition interpreter**, which is responsible for **generating bytecode** based on the AST that it received.

### Viewing Bytecode

We can actually see the bytecode that gets generated with the `--print-bytecode` flag in Node.

**Example**: Bytecode for our `calc` function:

```javascript
function calc(obj) {
  return obj.x * 50 + obj.z * obj.y;
}

calc({ x: 10, y: 20, z: 30 });
```

### Understanding the Bytecode

This may seem like a lot of data, but there are actually only two parts that are really important:

#### 1. Registers

Ignition uses **registers** in order to execute the bytecode:
- **Regular registers**: `r0`, `r1`, etc.
- **Accumulator register**: Used by bytecodes for their input and output (or both)
- **Argument registers**: `a0`, `a1`, etc. - used for values passed to the function

In this case, we passed an object containing `x`, `y`, and `z` keys to the function.

#### 2. Shape Table

The second important part: **`a0` points to a shape table** that contains information on where to find those properties on the object that we pass to the function.

### Bytecode Execution Walkthrough

Let's see what those bytecodes actually do:

**Line 1**: `LdaNamedProperty a0, [0], <feedback slot>`
- `Lda` specifies that a value gets **loaded into the accumulator**
- The value is the named property from the object we passed (stored in `a0`)
- The property can be found on **index 0**
- Index 0 maps to `x`
- We load the value of the `x` property (numeric value `10`)

**Line 2**: `Mul [50]`
- Multiply the current value in the accumulator by the small integer `50`
- Result: `10 * 50 = 500`

**Line 3**: `Star r0`
- The current value of the accumulator has to get stored in register `r0`
- `r0` now contains `500`

**Line 4**: `LdaNamedProperty a0, [2]`
- Again load a property and store into the accumulator
- This time from the **second index** which points to `y`
- `y` has a value of `20`
- Accumulator is now `20`

**Line 5**: `Star r2`
- Current value of the accumulator stored in register `r2`
- `r2` now contains `20`

**Line 6**: `LdaNamedProperty a0, [3]`
- Load named property from `a0` into the accumulator
- Value on the **third index** maps to `z`
- Has a value of `30`
- Accumulator is now `30`

**Line 7**: `Mul r2`
- Multiply the current value of the accumulator with the value stored in register `r2`
- `30 * 20 = 600`

**Line 8**: `Add r0`
- Add the value stored in register `r0` to the current value of the accumulator
- `600 + 500 = 1100`

**Line 9**: `Return`
- Return the value of the accumulator: `1100`

## Optimizations

### Initial Optimizations

The bytecode generated by the bytecode generator goes through some smaller optimizations, after which the bytecode actually gets **executed**.

It's possible to run this on our machines, so finally we have something that our machines can work with.

## V8 Shape Optimization

When we pass an object to V8 (such as the `{x, y, z}` object), it creates a **shape** for that specific object structure.

### What is a Shape?

In documentation or blog posts, this is also referred to as:
- Hidden class
- Map

**Note**: This is confusing because we also have classes and maps in JavaScript, but it's **not** a JavaScript class or JavaScript map. "Shape" is the way to go because we don't have those natively in JavaScript.

### Shape Definition

A **shape** is basically just the structure of that object.

The shape contains **pointers to the offsets** on which we can find the values of the properties on the object.

Even though we only specify the `x`, `y`, and `z` properties, there are many more built-in properties on objects that also all have their location stored somewhere in memory.

### Property Access Optimization

When we're trying to access a property on the object (for example `x`):
1. It checks: "Does this object have the same shape?"
2. "Yeah it does, okay cool"
3. "Now I want to get `x`, so I know the offset"

It can get it quicker by using the shape.

## Inline Caching

Shapes are really useful for an optimization technique that V8 uses, namely **inline caching**.

### What is Inline Caching?

With inline caching, we basically **store the results from previous operations** so that the next time we call the exact same operation, we already know the result.

Each time we do a property lookup, it can simply store the results for the offset from the last lookup.

In the future, when we're trying to perform the exact same action, it can simply get the result from the inline cache instead.

### Benefits Beyond the Interpreter

Inline caches are not only beneficial for the interpreter, but they also generate **really valuable feedback for the TurboFan optimizer**.

## Feedback Vector

Finally, we can go back to the bytecode example because these values are actually **references to a feedback vector slot** where it stores information about the execution of the function.

This includes information from arithmetic operations, such as the fact that **so far we've only added numbers** which result in a numeric value.

### Example: Plus Operator

One useful example: In JavaScript, you can also **concatenate strings with the plus operator**, which would have to be handled way differently internally.

But so far it knows: "Okay, I've only had numerical values, that's fine."

## TurboFan Optimizer

### Hot Functions

Let's say that we're invoking the `calc` function **hundreds of times**. This function is now considered **hot**.

Although the bytecode is already really fast, V8 actually uses **TurboFan (the optimizer)** in order to generate machine code that's even faster.

### Machine Code Generation

Based on:
- The bytecode
- The generated feedback for specific code blocks

TurboFan can generate **optimized architecture-specific machine code** that can run directly on your machine.

The next time that we invoke the function, it can just **skip over the bytecode** and immediately execute the machine code instead.

## The Problem: Dynamically Typed Objects

However, there is one problem in JavaScript: It's **dynamically typed**.

### Shape Changes

We can invoke the `calc` function with the same object hundreds and thousands of times, but there is **absolutely no guarantee** that this will always be the case in the future.

**Example**: We can also invoke the `calc` function with:
- An empty object `{}`
- Just an `x` key `{x: 10}`
- Just `x` and `y` keys `{x: 10, y: 20}`

For all those different types of objects, V8 generates a **new shape** that contains the new different properties.

### Inline Cache Updates

Previously we saw that the inline cache contained:
- A field with the value of the shape of the object
- The corresponding offset

However, if we passed multiple objects (so multiple shapes got generated), we also have to **update the inline cache** to point to multiple shapes and their offsets.

Now when we're trying to load a property from a specific object:
1. It first has to walk through all the possible shapes
2. In order to find the object that contains that specific property
3. This could result in a **linear search**, which is not very optimal

## Deoptimization

Previously we generated machine code for the `calc` function when it's only been invoked with one type of object (the object with `x`, `y`, and `z` keys).

However, if we call the `calc` function again but with a **different shape**:
1. TurboFan's **shape check fails**
2. We can no longer use this optimized machine code
3. We actually have to **deoptimize back** to the generated bytecode

**This is a pretty expensive operation that you mostly want to avoid.**

### After Deoptimization

The inline cache of the `calc` function is also updated to say: "Hey, it's actually got multiple shapes now."

The `calc` function can get hot and optimized again even after deoptimization, although TurboFan has to handle it a little bit differently when it's encountered multiple shapes.

## Inline Cache States

Inline caches actually have multiple states:

### 1. Monomorphic (Best Case)

If an inline cache has only seen **one type of object**, it's considered **monomorphic**.

This is pretty much the **best case scenario** because in that case we can:
- Generate optimized machine code
- Assume that in the future this function will get invoked with the same object shape

### 2. Polymorphic

If a cache has seen **two to four different shapes**, it's considered **polymorphic**.

### 3. Megamorphic (Worst Case)

If we're continuously invoking it with random types, it's considered **megamorphic**.

In this case: "You know what, never mind, I won't try to optimize it."

## Conclusion

Although it's pretty nice for us sometimes as developers that JavaScript is dynamically typed, it's not so great for the compiler.

It can really only work with **speculations** and just assume that in the future we will use the same type of data.
